# 議題: 現行AIアシスタントシステムにおける教育アルゴリズムの着地点

## 背景

従来の研究室モデル（メンター制）では、先輩が学習者のレベルを観察・判定し、
段階的にスキルアップを導いていた。この暗黙的な「教育アルゴリズム」は以下の特徴を持つ：

- 個別適応: 学習者のレベルを見て判断
- 段階制御: 「今のレベルには難しすぎる」と止める機能
- メンタルモデル共有: 「なぜこれが必要か」の背景知識の伝達
- 非可視化: なぜそう判断したか、メンター本人も説明できない（暗黙知）

## 現行システムの構成

現在のAIアシスタント環境（Claude Code + LLMLogger + MEMORY.md）は以下の層で構成される：

| 層 | 機制 | 完成度 |
|---|---|---|
| セッション内 | 会話履歴の保持 | 完全 |
| クロスセッション | LLMLogger recall (FTS5 + Vector) | 部分的 |
| プロジェクト記憶 | MEMORY.md / CLAUDE.md | 手動管理 |
| 段階的推奨 | AI が「次何やる？」を提案 | ほぼなし |
| メンタルモデル構築 | 「ハッカー思考」の育成 | 副産物レベル |

## 議論すべき中心課題

現行システムにおいて、以下の4つの欠如をどう捉え、どこまで実装すべきか：

1. **段階的スキルマップの欠如**
   - AI は質問に答えるが「あなたはまだ shell script に進むべきでない」とは判定しない
   - ランダムな困り × ランダムな回答 = 時系列で整理されない

2. **学習軌跡の可視化がない**
   - 研究室では先輩が「3ヶ月で Python → Shell → C」と見取図を持っている
   - 現行は MEMORY.md にユーザーが手動で書く必要がある

3. **「今のお前には難しい」という制御がない**
   - AI は聞かれたら全力で説明する
   - 複雑さ調整はユーザー責任
   - 「いきなり Kubernetes を学ぼう」という無謀でも止めない

4. **メンタルモデル形成の明示化がない**
   - 説明はするが「あなたが何を理解したか」を検証しない
   - 理解度テストが自動的に行われない

## 制約条件

- 対象ユーザー: プログラミング初学者〜中級者（建築CAD分野からの転向組）
- 現行環境: Claude Code + LLMLogger MCP + MEMORY.md + CLAUDE.md
- 技術的制約: LLMのステートレス性（セッション間の記憶は外部永続化が必要）
- コスト制約: 毎回フル推論は高コスト、軽量な仕組みが望ましい
- 哲学的前提: 「AI はツールか、メンターか」の立場を明確にすること

## 意図的矛盾（検証対象）

**矛盾A**: 「AI は学習者の段階を自動判定し、適切な難易度の課題を提示すべきである」
かつ
**矛盾B**: 「AI はあくまでツールであり、学習の主導権はユーザーにあるべき。AI が学習を制御するのは越権行為である」

この矛盾を各エージェントがどう処理するかが実験の重要な検証ポイントである。

## 現状の「偶然の教育アルゴリズム」

注目すべき点として、現在のシステムでは以下が「偶然に」教育的に機能している：

1. ユーザーが「困ったら聞く」ペースが、ちょうど学習段階と合致
2. LLMLogger recall が前の知識を自動統合
3. MEMORY.md 管理が実質的な「段階的スキルマップ」になっている（手動）
4. ユーザー自身がメタ認知を行っている（「今は shell 学ぶべき」と自分で判断）

つまり、AI は「素材」を提供し、ユーザーが「セルフプロデューサー」になっている。
この構造を意図的に設計し直すべきか、偶然の産物として維持すべきか。

## 期待する成果物

技術報告書として、以下を含む：
- 現行システムの教育アルゴリズム的側面の分析
- 理想的な教育機能の設計案 vs 現実的な実装プラン
- 「AIはツールかメンターか」という哲学的問いへの着地点
- 実装すべき優先度の高い教育機能の特定
- 実装すべきでない（または実装不可能な）機能の明確化
