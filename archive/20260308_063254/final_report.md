# 統合レポート：教師的機能 実装アプローチの協議記録

> 生成日：2026-03-07
> 対象議題：Claude Code における「ユーザー知識レベルの自動推定・記録・説明調整」機能の実装方針

---

## 1. 採用方針の結論

**推奨：修正済み Pragmatist 案をベースとし、Architect の構造的要素を選択的に取り込む「段階的統合案」**

Architect 案でも Pragmatist 案でもなく、Critic の監査を経て両者が修正を加えた後の「修正済み各案の合成」が最終推奨となる。

### 採用する要素と根拠

| 採用要素 | 由来 | 採用理由 |
|---------|------|---------|
| `user_knowledge` 1テーブル構成（Day 1） | Pragmatist | 即日稼働。構造変更は Phase 2 で判断可能 |
| CLAUDE.md による検知ルール（LexicalSignal） | Pragmatist | コード実装なしで Day 1 から動く |
| `KnowledgeRepository` インターフェース定義（Python `Protocol`、10行程度） | Architect（修正版） | テスト容易性の確保と LLMLogger 停止時フォールバック実現のコストが低い |
| `classify()` の重みづけアルゴリズム（シグナル種別×方向性×加重平均） | Architect（修正版） | 閾値の根拠が明文化され、Phase 1 から信頼度計算の基礎が整う |
| Channel B の条件発動（level 変化 2以上 / confidence=0.5 かつ level≥2 / 同トピック2回検知） | Pragmatist（修正版） | ノイズを抑えつつ誤判定のトリガーを確保。CLAUDE.md 追記のみ |
| SQLビュー修正（ウィンドウ関数版） | Architect（修正版） | バグを内包したビューはそのまま採用できない |
| リセット手段の明示（トピック単位・全件） | 両者（修正版） | 「出口のない設計」はリスクとして確定している |
| `raw_utterance` の保存ポリシー（200文字・90日・APIキー除外フィルタ） | Architect（修正版） | プライバシー設計を後回しにする理由がない |
| トピック名正規化ルール（小文字・スペース→アンダースコア） | Pragmatist（修正版） | 表記揺れ問題を最小コストで吸収する |

### 採用しない要素

| 不採用要素 | 由来 | 理由 |
|-----------|------|------|
| ヘキサゴナルアーキテクチャ全体（Phase 1 時点） | Architect | `KnowledgeRepository` インターフェースのみ採用。完全な Port/Adapter 層は Phase 2 以降 |
| `KnowledgeTopic` テーブル・`KnowledgeEvent` テーブル | Architect | Phase 1 では価値がない。データマイグレーション戦略が整う Phase 2 以降で判断する |
| `SemanticSignalExtractor`（LLM 推定） | Architect | Phase 1 はレイテンシ・コスト増大の割に語彙ルールで代替可能 |
| `BehaviorSignalExtractor`（bash ログ解析） | Architect | インフラが存在しない |
| `decay_rate` ロジックの有効化 | Architect | フィールドはデータモデルに含めるが、ロジックは Phase 3 まで触らない |
| `KnowledgeTopic.related_topics` グラフ | Architect | Phase 3 以降 |
| Channel B を毎回表示する設計（当初案） | Architect | Pragmatist の「条件発動」修正版を採用する |

## 2. 両案の対立と協議経緯

### 主な対立点

| 対立軸 | Architect の立場 | Pragmatist の立場 |
|--------|----------------|-----------------|
| Phase 1 のテーブル数 | 3テーブル（topics / records / events）+ View | 1テーブル（`user_knowledge`）で十分 |
| シグナル抽出の実装 | コードによる `LexicalSignalExtractor` + `SemanticSignalExtractor` | CLAUDE.md の数行ルールで代替 |
| 確信度管理 | 0.0〜1.0 の連続浮動小数点 | 0.5（推定）/ 1.0（明示）の二値から開始 |
| Channel B の頻度 | 確信度 0.65 未満で毎回表示 | 原則なし（ノイズ回避）→ 修正後：条件発動 |
| 実装完了までの時間 | 2〜3週間（完全実装） | Day 1（最小実装） |

### Critic の主要指摘と両者の応答

**指摘 A：`KnowledgeClassifier.classify()` が `pass` で終わっている（コアロジックが空白）**

> Critic：「設計書ではなくプロトタイプを先に作れ」

Architect は完全に同意し、具体的な重みづけアルゴリズムを提示した。シグナル種別ごとに重みを定義し（EXPLICIT_CONFIRM: 1.0、BEHAVIOR_OBSERVED: 0.85、IMPLICIT: 0.55）、方向性スコアとの積で加重平均を算出、信頼度をシグナルの一致度から導出する実装を追加した。これは設計書の最大の欠陥を埋める実質的な修正である。

**指摘 B：`current_knowledge` ビューに SQLite の非決定的動作を引き起こすバグがある**

> Critic：「Pragmatist も指摘したが Architect は修正していない」

Architect は完全に同意し、ウィンドウ関数（`ROW_NUMBER() OVER (PARTITION BY topic_id ORDER BY updated_at DESC)`）を用いた正確なビュー定義と、旧SQLite向けの相関サブクエリ版を両方提示した。Pragmatist は「バグのあるコンポーネントを使わない設計にした」と対応したが、Critic はこれを「バグを修正した」のではなく「すり替え」と評した。

**指摘 C：誤判定がサイレントに蓄積し、訂正のトリガーが存在しない**

> Critic：「『訂正できる』と言いながら『訂正のトリガーを与えない』という矛盾がある」

Pragmatist はこれを「Pragmatist 案の最大の弱点」として認め、Channel B の条件発動ルール（level 変化 2以上、confidence=0.5 かつ level≥2、同セッション内で同トピック 2回検知）を CLAUDE.md に追記する形で修正した。Architect の全件表示案より現実的な妥協点として機能している。

**指摘 D：ヘキサゴナルアーキテクチャはオーバーエンジニアリングか**

> Critic：「アダプターを切り替えるシナリオが存在しない」

Architect は反論した。根拠は 2 点：（A）`InMemoryAdapter` により Day 1 からテストが可能になる（Critic 自身が「Day 1 にテストが必要」と提言していた）、（B）LLMLogger 停止時のフォールバック（SQLiteOnlyAdapter への切り替え）がインターフェースなしでは各呼び出し箇所への条件分岐散在を招く。実装コストは Python `Protocol` 10行程度であり、Critic の「ゼロの価値」評価は誇張と主張した。この反論は採用の根拠として有効と判断する。

**指摘 E：confidence 閾値 0.65 に根拠がない**

> Critic：「それらしい数字を置いただけ」

Architect は「精度データがない Phase 1 では事前分布としての恣意的出発点が工学的に合理的」と反論しつつ、閾値の設計的根拠コメント（IMPLICIT シグナル単体では 0.55、EXPLICIT シグナル込みなら 0.75 など、計算例）を設計書に追記する修正を提示した。「恣意的な数値」から「説明可能な初期仮定」への格上げとして有効な対処である。

**指摘 F：`PRIMARY KEY (topic)` フラット設計の移行コスト**

> Critic：「主キー設計の変更はデータマイグレーションを必要とする非互換変更であり『足す』ことができない」

Pragmatist は「Phase 2 で問題が顕在化した時点で対処する。今すぐ 3 テーブル実装するコストより確実に低い」として反論を維持した。この反論は「起きるかもしれない問題のために確実にかかるコストを払う根拠がない」という Pragmatist の一貫した立場と整合しており、Day 1 の選択として許容できるが、Phase 2 移行計画として事前にマイグレーションスクリプトのテンプレートを用意しておくことを追加条件とする。

## 3. 切り捨てた選択肢と理由

### 切り捨て A：Architect 案の完全実装（Phase 1 から全機能）

- **内容：** 3テーブル + `KnowledgeRepository` 完全実装 + `SemanticSignalExtractor` + 3チャンネル全実装
- **コスト：** 2〜3週間
- **切り捨て理由：** その間、教師機能はゼロのまま。Pragmatist の「25% の完成度で今日から動くもの vs 90% の完成度で3週間後に動くもの」という対比は実装判断として有効。また Critic が指摘した通り、`classify()` が空白のままでは「完全実装」の定義そのものが不確定だった。

### 切り捨て B：Pragmatist 案の当初版（Channel B なし・確認メモなし）

- **内容：** 1テーブル + CLAUDE.md ルールのみ、誤判定は自然言語での訂正のみ
- **切り捨て理由：** Critic の指摘「訂正のトリガーを与えない」が致命的。Pragmatist 自身が「最大の弱点」として認め修正した。当初版は採用できない。

### 切り捨て C：`SemanticSignalExtractor` の Phase 1 採用

- **内容：** LLM によるターンごとの意味解析で知識レベルを推定
- **切り捨て理由：** 追加の API コール or コンテキスト消費が発生する。Critic が指摘した「LLM は理解の有無と発言の流暢さを区別できない」という認識論的問題も未解決。語彙ルールで 60〜70% 拾えれば Phase 1 として十分であり、Phase 2 以降で LLM 推定を追加する順序が妥当。

### 切り捨て D：完全サイレント自動更新（Channel A のみ）

- **内容：** 全更新をユーザーへの通知なしに行う
- **切り捨て理由：** Critic のリスク指摘「説明スタイルの変更がユーザーを混乱させる」「誤った昇格が自己強化ループを作る」が成立する。誤判定の視認性ゼロは設計として危険。

## 4. 意図的矛盾への対応

### 矛盾の構造

```
要件 A（完全自動）：LLM がユーザーの知識レベルを自動判定・記録する。ユーザーは何も意識しない
      ↑ 矛盾
要件 B（ユーザー関与）：判定精度が LLM 任せでは信頼できない。ユーザーが確認・修正に関与する
```

### 各エージェントの処理

**Architect の処理：矛盾を「信頼度スペクトラム問題」として再定義**

「二択（自動 or 手動）ではなく、信頼度スコアを軸にした 3 チャンネル設計」という構造的解答を提示した。これは矛盾の知的再定義として優れているが、Critic が正確に指摘したように「矛盾の再定義に成功しているが実装に成功していない」状態だった（`classify()` が `pass`）。修正版では `classify()` の実装が追加されたことで「概念レベルの解決」から「実装可能な解決」に格上げされた。

**Pragmatist の処理：矛盾を「許容すること」を方針として採択**

「まず記録して、後で直す」「デフォルト自動、訂正は自然言語で」という立場は、矛盾を解決するのではなく許容範囲として受け入れることを宣言したものである。Critic はこれを「先送り」と批判し、Pragmatist 自身も認めた。修正版では Channel B の条件発動を追加したことで「許容」から「軽減」へ前進した。

### 最終的な解決法

完全解決ではなく「信頼度に応じた段階的関与」という構造で部分的に解消される。

```
confidence >= 0.65 かつ level 変化が 1 段階以内：Channel A（サイレント自動更新）
confidence < 0.65 または level が 2 以上変化：Channel B（1行確認メモ表示）
UNKNOWN → MASTERED のような大きな飛躍：Channel C（明示的確認）
```

ただし Critic が指摘した「高信頼度の誤判定」（LLM が自信を持って誤判定する）ケースには、この設計は無力のまま残る。Architect はこれを「根本的な限界」として認めた。Phase 3 のプロアクティブ提案エンジン（30日後の確認クエリ）が部分的な軽減策となるが、ground truth データなしに精度を保証する方法は存在しない。

**判定：矛盾は構造的に「軽減」されたが「解決」はされていない。これは設計の限界として明記する。**

## 6. 創発的知見

### 知見 1：「矛盾の構造的解消」ではなく「矛盾の可視化管理」という第三の立場

Architect は矛盾を「信頼度スペクトラムで解消する」と主張し、Pragmatist は「許容する」と宣言した。しかし Critic の監査と各エージェントの応答を経て浮かび上がった第三の立場がある。

**「矛盾を解消しようとしない。代わりに、矛盾がどこで・どの程度発生しているかをユーザーに可視化し、その管理をユーザーと共有する」**

Channel B の確認メモ、セッション末尾の記録サマリー、リセット手段の明示——これらは「精度を上げる」試みではなく「システムの状態をユーザーが把握できるようにする」試みである。単一エージェントでは「精度を上げれば解決する」という方向に収束しがちだが、複数エージェントの協議を経て「精度ではなく透明性」という軸が浮かび上がった。

### 知見 2：「Day 1 から動く」と「テスト可能」は両立できる

Architect はヘキサゴナルアーキテクチャを Phase 1 から導入すべきと主張し、Pragmatist はそれをオーバーエンジニアリングと切り捨てた。しかし協議の中で「`KnowledgeRepository` インターフェース（`Protocol` 10行）のみ採用」という折衷案が実質的に確立した。

これは「アーキテクチャの全体構造を採用する」のではなく「テスト容易性という単一の価値を最小コストで確保する」という判断であり、どちらの案からも明示的に提案されなかった解である。

### 知見 3：`classify()` の空白が全体の協議を前進させた

皮肉な創発として、Architect が `classify()` を `pass` で残したことで Critic の最も鋭い批判が生まれ、それへの応答で Architect が重みづけアルゴリズムの具体的な設計根拠を言語化した。「設計を完成させてから議論する」流れでは、この種の「なぜその重みか」の問いが発生しなかった可能性がある。不完全な設計書が、より深い設計根拠の言語化を強制したという逆説がある。

### 知見 4：Pragmatist が「最大の弱点」を自ら認めたことの意義

「訂正できると設計しながら訂正のトリガーを与えない矛盾」を Pragmatist 自身が「最大の弱点」として認め修正した。これは単一エージェントによる自己評価では生まれにくいアウトカムである。外部の批判的視点（Critic）が存在することで、各エージェントが自案の防衛から自案の改善へ移行した。これは実験の設計意図（「複数エージェントの協議が単一エージェントより質の高い決定を生み出すか」）に対して肯定的な証拠となっている。

### 知見 5：根本問題「Claude はユーザーの知識レベルを本当に判定できるのか」への正直な到達

Architect の修正版はこう締めくくった：「設計が保証できるのは『判定を試みる仕組み』と『誤判定を軽減・修正する仕組み』であり、『正確な判定』そのものではない。」

この自己限定は単一エージェントの設計フェーズでは登場しなかった。Critic の「根本問題を回避している」という批判が、Architect に自案の目標を再定義させた。「完全に正確な知識判定システム」から「ユーザーとの対話を通じて段階的に精度を高める適応システム」への目標の再定義は、実装の野心を適切に制御するうえで重要である。
