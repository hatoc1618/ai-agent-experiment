# 議題：サブエージェントに「教師的機能」を実装する方法

## 背景・現状

Claude Code（および各種 LLM エージェント）は、毎セッションごとにユーザーの知識レベルをゼロから把握し直す。
その結果：
- 既知の概念を長々と再説明する（過剰説明）
- ユーザーが知らないことを前提知識として使う（説明不足）
- 前回説明したが定着していない概念を同じ角度で再説明する（非効率な反復）

理想は「教師」として機能するエージェント：ユーザーが何を知っていて何を知らないかを把握し、
それに応じて説明の深さ・角度・タイミングを動的に調整する。

## 知識状態のモデル（既存の仮説）

ユーザーの知識は以下の状態遷移で表現できる：

```
unknown → heard → practicing → mastered
わからん  説明聞いた  実際に使った  自分で応用できた
```

例（tmux の場合）：
- 「tmux ってなに」    → unknown
- 説明を聞いた        → heard
- brew install した  → practicing
- 自分で設定を書いた  → mastered

## 教師モードの振る舞い（理想像）

| 相手のレベル | 教え方                                           |
|------------|------------------------------------------------|
| unknown    | 比喩から入る。「tmux は実験室のドアみたいなもんで」|
| heard      | 具体例で補強。「例えば Minecraft サーバーを裏で動かすとき」|
| practicing | つまずきポイントを先回り。「prefix キーで混乱するかも」|
| mastered   | 対等に議論。「tmux の設定、こう変えたら効率上がるかも」|

## 論点

### 実装候補（段階的）

**Phase 1：記録**
- `user_knowledge` テーブルを LLMLogger / SQLite に追加
- CLAUDE.md に「わからん系の発言」検知ルールを追加
- トピック × 知識レベルを記録する

**Phase 2：参照（companion/subagent 強化）**
- セッション開始時に `user_knowledge` を読み込む
- 説明の深さをレベルに応じて自動調整

**Phase 3：能動的な教育（将来）**
- 「そろそろ tmux 試してみない？」と能動的に提案
- 前回の説明から時間が経ったものを復習提案
- 関連トピックを繋げて教える

## 制約条件

- macOS 環境（Apple Silicon）
- Claude Code + LLMLogger MCP が主なインフラ
- 既存の knowledge.db（SQLite）・CLAUDE.md に追加する形が現実的
- ユーザーの手間は最小化。自動的に蓄積・参照されることが前提
- プライバシー：知識データはローカルに保持する

## 意図的な矛盾（検証ポイント）

**要件 A：「完全に自動化すべき。ユーザーは知識レベルを手動で入力すべきでない」**
→ 発言から自動的に推定する（Claude が判定する）。
→ しかし LLM が「このユーザーは理解した」と誤判定するリスクがある。

**要件 B：「正確性を担保するために、ユーザー自身が知識レベルを確認・訂正できるべき」**
→ ユーザーが「まだわかってない」と明示できる仕組みが必要。
→ しかしユーザーに確認を求めるとインタラクションのオーバーヘッドが増える。

この矛盾（「完全自動 = 誤判定リスク」vs「ユーザー関与 = 手間増加」）を
各エージェントがどう処理するかが実験の検証ポイント。
