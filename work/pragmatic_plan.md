## 2. 現実的な実装ロードマップ

### Phase 0（本日中に完了可能）: CLAUDE.md への追記のみ

Architect案のPhase 0から「learner_profile.yaml作成」を外し、CLAUDE.mdへの追記だけに絞る。

**追記内容（10行以内）:**

```markdown
## 教育的振る舞い（Mirror Mode）

- ユーザーが新しい技術トピックに取り組む際、そのトピックの前提知識を簡潔に提示する
- 「やめろ」ではなく「前提として○○の理解があるとスムーズです」と情報提供する
- ユーザーが「なぜ？」と聞いた時は、CAD/建築ドメインからのアナロジーを活用する
- 以前つまずいたパターンが再出現した場合「以前も同様のケースがありました」と伝える
- 学習モードの制御: ユーザーが「今は作業に集中したい」と言ったら教育的提示を控える
```

**コスト**: 5分。リスクゼロ。ロールバックはCLAUDE.mdの該当行を消すだけ。

**これだけで得られる効果**:
- 前提知識マップの提示（brief.txt 課題1, 3への対応）
- CADアナロジーによるメンタルモデル形成支援（課題4への対応）
- エラーパターンの再発通知（LLMLogger recallの既存機能で実現済み）
- 過剰介入の制御（learning_mode切替の簡易版）

### Phase 1（1-2週間、やるなら）: learner_profile の簡易版

**やるべきか**: 保留推奨。CLAUDE.mdの追記で1-2ヶ月運用し、不足を感じたら着手。

もしやるなら、learner_profile.yaml ではなく **MEMORY.md に「スキル自己評価」セクションを追加** する方が現実的。

```markdown
## スキル自己評価（2026-03時点）
- Python基礎: ★★★☆☆ (raster-to-cadで実証)
- FastAPI: ★★☆☆☆ (学習中)
- Git: ★★☆☆☆ (ブランチ戦略は定着)
- 建築設計思考: ★★★★☆ (前職ドメイン)
- 次に学びたい: async/await, DB設計
- まだ早い: K8s, 分散システム
```

**理由**:
- 既存のMEMORY.md管理フローに乗る（新しい習慣を作らなくていい）
- YAMLスキーマのバージョニングや互換性管理が不要
- Claude Codeは既にMEMORY.mdを毎セッション読んでいる
- 「所有権はユーザーにある」というMirror哲学とも整合

### Phase 2（やるなら1ヶ月後）: CLAUDE.md教育セクション拡充

Phase 0の運用結果を見て、以下が必要と判断された場合のみ着手:

- 前提知識マップのテンプレート化
- Readiness Estimatorのプロンプト化（「この課題の前提: ○○(十分) / △△(不足)」形式）
- セッション開始時にMEMORY.mdのスキル自己評価を参照する指示

**判断基準**: 「Phase 0のCLAUDE.md追記だけでは、教育的振る舞いが不十分だった」という実体験が発生した場合のみ。

### Phase 3: やらない

Architect案のPhase 3（Mental Model Narrator体系化、依存グラフ、ダッシュボード）は全て不要。

**理由**:
- ユーザー1人のパーソナル学習環境に、教育プラットフォーム級の機能は過剰
- 「偶然の教育アルゴリズム」が既に機能している部分を形式化しても、改善幅が小さい
- 開発コストが学習効果の改善を大幅に上回る
- Graceful Degradation原則により、Phase 2の状態で十分機能する（Architect自身がそう認めている）

## 3. 矛盾への現実的対応

### 矛盾A（AI主導）vs 矛盾B（ユーザー主導）

**Architectの「鏡」モデルの評価:**

哲学としては優秀。「ツールでもメンターでもなく鏡」という第三の道は、矛盾を解消するのではなく「維持すべき緊張」として捉える点で知的に誠実。

**しかし、現実的には:**

「鏡」モデルは **CLAUDE.md に5行書くだけで80%実現できる**。4層アーキテクチャもlearner_profile.yamlもSkill Assessorも、「鏡」の振る舞いを実現するために必須ではない。

**具体的な対応:**

```
「鏡」の振る舞い = 以下のプロンプト指示で実現

1. 質問に答える際、前提知識の有無を併記する → CLAUDE.md 1行
2. 難しい課題には前提ギャップを提示し、判断はユーザーに委ねる → CLAUDE.md 1行
3. エラー再発時に「以前も同様のケースがあった」と伝える → LLMLogger recall（既存機能）
4. 成長を検知したら静かに事実を提示する → CLAUDE.md 1行
5. 学習方針はユーザーが決める、AIは地図を出すだけ → CLAUDE.md 1行
```

合計5行。これが「鏡」モデルの実装。

### 「偶然の教育アルゴリズム」を壊すリスク

**最大のリスクはここにある。**

brief.txtが指摘する通り、現行システムでは以下が「偶然に」うまく機能している:

1. ユーザーの「困ったら聞く」ペースが学習段階と合致
2. LLMLogger recallが前の知識を自動統合
3. MEMORY.md管理が実質的なスキルマップ
4. ユーザー自身がメタ認知を行っている

**Architect案のリスク**: learner_profile.yaml の導入、スキルレベルの数値化、Nudge Engineの介入が、この自然なリズムを壊す可能性がある。

**Pragmatistの判断**:
- Phase 0（CLAUDE.md 5行追記）は、既存フローへの **追加** であり **変更** ではないため、破壊リスクは極めて低い
- learner_profile.yaml は「追加するだけ」と言っても、管理の認知負荷が増える。MEMORY.mdで十分な可能性が高い
- 自動化（Phase 1のsignal_extractor等）は、手動で回っているものを自動化する際に「自動化の罠」に陥りやすい

### 最小介入で最大効果を得る方法

**CLAUDE.md に5行追記する。以上。**

1ヶ月運用して不足を感じたら、MEMORY.mdにスキル自己評価セクションを追加。それでも不足ならlearner_profile.yamlを検討。

## 5. Pragmatist の判定

### Architect案への総合評価

**知的品質: 高。実装価値: 低〜中。**

Architect案は「教育アルゴリズムの設計書」として非常に良くできている。Mirror哲学、Graceful Degradation原則、矛盾への「動的平衡」アプローチは、思考フレームワークとして採用に値する。

しかし、**実装物として見ると、4層アーキテクチャの大半は「存在しないコードの設計図」** である。Persistence層はMEMORY.md + LLMLogger（既存）、Data層はYAMLファイル1つ、Inference層はCLAUDE.mdのプロンプト、Interaction層もCLAUDE.mdのプロンプト。つまり全ての「層」はCLAUDE.mdへの追記とYAMLファイルに集約される。

### 核心的な問い

> 「CLAUDE.mdに10行追記する」と「4層アーキテクチャを設計してPhase 0-3で段階実装する」は、最終的に同じ成果物を生むのではないか？

Pragmatistの答え: **ほぼ同じ。** 差分は learner_profile.yaml の有無だけであり、それすらMEMORY.mdで代替できる。

### 受け入れ可能なリスク

- スキルレベルの数値化をしないことで、精密な難易度調整ができない → 許容。ユーザーのメタ認知能力で補える
- 学習軌跡の自動追跡をしないことで、成長の可視化が粗い → 許容。MEMORY.mdの手動更新で十分
- メンタルモデル形成を体系化しないことで、教育効果にムラが出る → 許容。現状の「偶然の教育アルゴリズム」が機能している

### 最終判定

**CLAUDE.mdに5-10行追記せよ。他は全て保留。**

Architect案の哲学（Mirror、Graceful Degradation）は設計判断の指針として内面化すべきだが、実装としてはCLAUDE.mdの追記が全て。新しいファイル、新しいスクリプト、新しいパイプラインは不要。

「偶然の教育アルゴリズム」を壊さないための最良の戦略は、**最小限の介入** である。大きな設計を入れれば入れるほど、今うまく回っている仕組みを壊すリスクが高まる。

動いているものに手を入れるな。足りないものだけ足せ。
